1. Introduction

1.1 Design Overview

This Software Design Description (SDD) provides a comprehensive architectural and detailed design specification for the Code Quality & Fault Prediction System. It describes the system architecture, component design, data models, interfaces, algorithms, and user interface details necessary for developers to construct the system in accordance with the requirements specified in the SRS (Chapter 2).

This document conforms to IEEE 1016-2009 — IEEE Standard for Information Technology — Systems Design — Software Design Descriptions. All design decisions described herein are framed as planned specifications — the actual coding process has not yet commenced.

The design document shall serve as:

  • A blueprint for system implementation and construction
  • A reference for understanding system structure and component interactions
  • A guide for system maintenance and future enhancements
  • A communication tool between architects, developers, and stakeholders
  • A detailed specification for integration and testing activities

3. Composition Viewpoint

The Composition Viewpoint describes the overall system architecture, its decomposition into tiers and modules, and the rationale for technology selection.

3.1 System Architecture — Three-Tier Layered Architecture

Tier 1 — Presentation Layer: React 19 Frontend (Port 5173)

  • UI components built with React 19 + TypeScript 5.9 + Tailwind CSS 3.4
  • 40+ Radix UI accessible component primitives
  • Client-side state management: React Context API + Zustand 4.5
  • Data visualization: Recharts 3.6 (interactive charts)
  • Routing: React Router 6.26 (SPA navigation)

Tier 2 — Application / Business Logic Layer: Flask Backend (Port 5000)

  • RESTful API endpoints using Flask with Flask-CORS
  • Service modules: GitHubService, CodeAnalyzer, MLService
  • Authentication: Session-based with PBKDF2-SHA256 password hashing
  • ML Pipeline: scikit-learn Random Forest, Pandas feature engineering, Joblib persistence

Tier 3 — Data Access Layer: SQLite Database

  • SQLAlchemy ORM for database abstraction and SQL injection prevention
  • 7 tables: User, Settings, Repository, CodeMetric, Prediction, BugReport, TestCase
  • File-based storage: app.db (SQLite), repos/ (cloned repos), models/*.pkl (ML models)

Design Principles Applied:

  • Modularity: Each service (GitHubService, CodeAnalyzer, MLService) has single responsibility
  • Loose Coupling: Frontend communicates with backend only via REST API; no shared state
  • High Cohesion: Related functionality grouped in packages (services/, ml/, database/)
  • Separation of Concerns: UI (React), business logic (Flask services), data (SQLAlchemy ORM)
  • DRY: Reusable UI components (40+ Radix primitives), shared hooks, utility functions
  • SOLID: Interface segregation via separate service classes; dependency injection via Flask app context
