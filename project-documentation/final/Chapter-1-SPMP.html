1. Overview

1.1 Project Summary

1.1.1 Purpose

This Software Project Management Plan (SPMP) defines the project management approach for the Code Quality & Fault Prediction System — an intelligent web-based platform that leverages machine learning to analyze code repositories, predict fault-prone modules, and provide actionable recommendations for software quality improvement.

This document shall serve as a reference guide for the development team, stakeholders, and project managers throughout the software development lifecycle. It establishes project objectives, scope, deliverables, organizational structure, risk management strategies, and a roadmap for project execution and monitoring in accordance with IEEE 1058-1998.

1.1.2 Scope

Project Name: Code Quality & Fault Prediction System

The system shall provide an end-to-end solution for code quality analysis and fault prediction by integrating with GitHub repositories. It shall analyze code complexity, predict bug-prone areas using machine learning, and offer intelligent recommendations for test prioritization and code improvements.

Key Deliverables:
  1. Web-based frontend application
  2. RESTful API backend service
  3. Machine Learning prediction engine
  4. SQLite database with 7 normalized tables for data persistence
  5. GitHub integration service for repository ingestion and metadata
  6. User authentication and session management system
  7. Real-time analytics dashboard with interactive visualizations
  8. Comprehensive documentation set (SPMP, SRS, SDD, STD)

1.1.3 Objectives

  1. Develop a functional web application that integrates with GitHub for automated code analysis
  2. Implement a machine learning pipeline capable of predicting fault-prone modules with ≥ 75% accuracy
  3. Provide actionable visualizations and metrics dashboards for code quality assessment
  4. Deliver complete IEEE-standard documentation (SPMP, SRS, SDD, STD) before implementation begins
  5. Complete the project within the planned 30-week timeline (November 2025 – June 2026) with iterative milestones

3. Managerial Process


3.1 Management Objectives and Priorities

Primary Objectives:

  1. Quality: Deliver a system that meets all functional and non-functional requirements specified in the SRS, with ML prediction accuracy ≥ 75%
  2. Timeliness: Complete the project within the 30-week planned schedule (Nov 2025 – Jun 2026), meeting all milestone deadlines
  3. Documentation: Produce IEEE-compliant documentation (SPMP, SRS, SDD, STD) before implementation begins
  4. Integration: Achieve seamless integration between frontend, backend, and ML components




Priority Ranking:

3ScheduleTimely delivery is important for academic submission

3.2 Assumptions, Dependencies, and Constraints

Assumptions:

  1. The sole developer has the requisite skills in React, Python, Flask, and ML
  2. GitHub API shall remain available and stable throughout the project
  3. Internet connectivity shall be available for GitHub API access during development and testing
  4. SQLite shall be sufficient for the expected data volume (up to 10,000 files per repository, 100 users)
  5. Open-source tools (Radon, Lizard, scikit-learn) shall provide accurate metrics and predictions
  6. Modern web browsers (Chrome 100+, Firefox 100+, Safari 15+, Edge 100+) shall be used by end users

Dependencies:

DependencyTypeImpact if Unavailable
GitHub REST API v3External ServiceRepository data cannot be fetched; core functionality blocked
Python 3.8+ RuntimePlatformBackend and ML components cannot execute
Node.js 18+ RuntimePlatformFrontend cannot be built or served
scikit-learn LibraryLibraryML training and prediction pipeline unavailable
Radon / LizardLibraryCode complexity metrics cannot be calculated



Constraints:

  • Technology: Frontend must use React/TypeScript; Backend must use Python/Flask; Database must be SQLite
  • API Rate Limits: GitHub API limited to 5,000 requests/hour (authenticated)
  • Performance: Repository analysis shall complete within 5 minutes for repositories with < 1,000 files
  • Deployment: Single-server local deployment model; no cloud infrastructure
  • Security: Passwords shall be hashed (PBKDF2); GitHub tokens shall be stored securely

4.1.2 Development Methods

  • Agile Practices: Iterative development with 1-2 week sprints, self-managed task tracking
  • Code Reviews: Self-review of code against coding standards before committing
  • Branching Strategy: Feature branches off main; merge after self-review
  • Coding Standards: PEP 8 for Python, ESLint for TypeScript, consistent naming conventions
  • Documentation-First: IEEE-compliant documents (SPMP, SRS, SDD, STD) shall be completed before implementation begins

4.3 Project Support Functions

4.3.1 Configuration Management

  • All source code shall be version-controlled using Git
  • Configuration files (.env) shall be excluded from version control; templates provided
  • Database schema changes shall be tracked and documented

4.3.2 Quality Assurance

  • Self code reviews shall be performed before committing changes
  • Automated testing shall cover critical paths (unit + integration tests)
  • Code coverage target: ≥ 70% for backend services
  • Static analysis using ESLint (frontend) and Radon (backend)

4.3.3 Problem Resolution

  • Issues shall be tracked using GitHub Issues with labels (bug, feature, documentation)
  • Bugs shall be prioritized: Critical (fix immediately), High (fix within 1 day), Medium (fix within sprint), Low (backlog)
  • Escalation path: Developer → Academic Supervisor
