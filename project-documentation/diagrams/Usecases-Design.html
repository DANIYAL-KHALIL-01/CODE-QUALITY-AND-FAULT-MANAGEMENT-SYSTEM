1. Use Cases

This section provides detailed use case specifications for core system operations.

UC-01: User Registration

Primary Actor: User

Goal: Create a new user account

Preconditions: User has not registered; system is available

Postconditions: User account created; user can log in

Trigger: User clicks "Sign Up"

Main Flow:

  1. User opens the registration page
  2. User enters username, email, and password
  3. System validates input
  4. System stores user account (hashed password)
  5. System confirms registration

Alternate Flows:

  • AF-1: Invalid input → show validation errors
  • AF-2: Username/email already exists → show error

UC-02: User Login

Primary Actor: User

Goal: Access the system dashboard

Preconditions: User account exists

Postconditions: Session created; dashboard accessible

Trigger: User clicks "Login"

Main Flow:

  1. User enters credentials
  2. System validates credentials
  3. System creates session
  4. User is redirected to dashboard

Alternate Flows:

  • AF-1: Invalid credentials → show error

UC-03: Add Repository

Primary Actor: User

Goal: Register a GitHub repository for analysis

Preconditions: User logged in; valid repository URL

Postconditions: Repository stored and visible in list

Trigger: User clicks "Add Repository"

Main Flow:

  1. User enters repository URL
  2. System validates the URL
  3. System fetches repository metadata from GitHub
  4. System saves repository details
  5. Repository appears in user list

Alternate Flows:

  • AF-1: Invalid URL → show error
  • AF-2: GitHub API failure → show error

UC-04: Analyze Repository

Primary Actor: User

Goal: Generate code metrics and predictions

Preconditions: Repository exists and is accessible

Postconditions: Metrics and predictions stored

Trigger: User clicks "Analyze"

Main Flow:

  1. System clones repository
  2. System computes code metrics
  3. System runs ML predictions
  4. System saves results
  5. User views analysis results

Alternate Flows:

  • AF-1: Clone fails → show error
  • AF-2: Analysis fails → show error

UC-05: View Dashboard

Primary Actor: User

Goal: Visualize metrics and risks

Preconditions: Analysis data exists

Postconditions: User views summary insights

Trigger: User navigates to dashboard

Main Flow:

  1. System loads dashboard data
  2. System displays charts and cards
  3. User explores metrics

UC-06: Manage Bug Reports

Primary Actor: User

Goal: Create and track bugs

Preconditions: User logged in

Postconditions: Bug record saved and visible

Trigger: User clicks "Report Bug"

Main Flow:

  1. User submits bug details
  2. System validates inputs
  3. System stores bug report
  4. Bug appears in list

3. Design Description

This section summarizes the high-level design of the Fault Prediction System.

3.1 Architecture Overview

Pattern: Three-tier layered architecture

  • Presentation Layer: React + TypeScript frontend for UI and visualization
  • Application Layer: Flask backend handling APIs, business logic, and ML orchestration
  • Data Layer: SQLite database for persistence and repository analytics

3.2 Core Components

Component Responsibility
GitHubService Repository metadata retrieval and cloning
CodeAnalyzer Code metric calculation (CC, MI, Halstead, churn)
MLService Feature extraction, model prediction, risk classification
Auth Service User registration, login, session handling

3.3 Data Model Summary

  • users: authentication and profile data
  • settings: user preferences and tokens
  • repository: GitHub repository metadata
  • code_metric: metrics per file
  • prediction: ML predictions per file
  • bug_report: bug tracking
  • test_case: prioritized test cases

3.4 Interfaces

Frontend ↔ Backend: REST API over HTTPS with JSON payloads

Backend ↔ GitHub: GitHub REST API v3 (authenticated tokens)

Backend ↔ Database: SQLAlchemy ORM with SQLite

5. Sequence Diagram Notation Guide

Standard UML Elements Used:

Element Representation Purpose
Actor Stick figure External user or system initiating actions
Participant Rectangle box at top System component or object
Lifeline Vertical dashed line Object's lifetime during interaction
Activation Box Thin rectangle on lifeline Period when object is active/processing
Synchronous Message Solid arrow → Method call or request
Return Message Dashed arrow ⟵ Response or return value
Alt Fragment Rectangle with "alt" Conditional logic (if-else)
Loop Fragment Rectangle with "loop" Iterative processing
Par Fragment Rectangle with "par" Parallel execution
Note Folded rectangle Additional information/comments
