1. Introduction

1.1 Purpose

This Software Requirements Specification (SRS) document specifies the functional and non-functional requirements for the Code Quality & Fault Prediction System. It is intended for developers, testers, project managers, stakeholders, and maintenance teams to understand system capabilities, constraints, and quality attributes in accordance with IEEE 29148-2018 (superseding IEEE 830-1998).

This document shall serve as a baseline for software design, implementation, and testing activities. All system features described herein are framed as planned capabilities — the actual coding process has not yet commenced.

1.2 Scope

Product Name: Code Quality & Fault Prediction System

The system shall provide an end-to-end solution for code quality analysis and fault prediction by integrating with GitHub repositories. It shall analyze code complexity, predict bug-prone areas using machine learning, and offer intelligent recommendations for test prioritization and code improvements.

Major Capabilities:

  1. User Authentication & Management — Secure registration, login, profile management, GitHub token integration
  2. Repository Analysis — GitHub repository import, multi-language code analysis, complexity metrics calculation
  3. Machine Learning Predictions — Fault-prone module identification, risk scoring, confidence levels
  4. Interactive Dashboards — Real-time metrics visualization, risk distribution charts, complexity trends
  5. Test Prioritization — ML-based test recommendations, risk-based test ordering
  6. Bug Report Management — Bug tracking, severity management, status workflow
  7. Recommendations Engine — Code improvement suggestions, refactoring opportunities

1.3 Product Overview

The Fault Prediction System is an intelligent web-based platform that analyzes software repositories to predict fault-prone modules using machine learning techniques. The system integrates with GitHub to extract repository data, calculate code complexity metrics (Cyclomatic Complexity, Maintainability Index, Halstead metrics, Lines of Code, Code Churn), and provide actionable insights for improving software quality and reducing technical debt.

2.4 Constraints

  • Technology Stack: Frontend shall use React 19 / TypeScript 5.9; Backend shall use Python 3.8+ / Flask; Database shall be SQLite
  • API Rate Limits: GitHub API limited to 5,000 requests/hour (authenticated)
  • Performance: Repository analysis shall complete within 5 minutes for repositories with < 1,000 files
  • Deployment: Single-server local deployment model; no cloud infrastructure required
  • Security: Passwords shall be hashed using PBKDF2; GitHub tokens shall be stored securely
  • Regulatory: GDPR-compliant data handling practices

2.5 Assumptions and Dependencies

Assumptions:

  1. Users shall have stable internet connectivity for GitHub API access
  2. GitHub API shall remain available and stable throughout the project lifecycle
  3. Users shall use modern web browsers (Chrome 100+, Firefox 100+, Safari 15+, Edge 100+)
  4. SQLite shall be sufficient for the expected data volume (up to 10,000 files per repository, 100 users)
  5. Python 3.8+ and Node.js 18+ runtimes shall be available in the deployment environment

4. Specific Requirements

4.1 External Interface Requirements

4.1.1 User Interfaces

The system shall provide a modern, responsive web-based user interface built with React 19 and TypeScript. The interface shall be designed for intuitive interaction across desktop and tablet devices (768px to 1920px width).

Key UI Components:
  • Dashboard: Overview of repositories, metrics summaries, and risk distribution
  • Repository Management: Add, list, analyze, and delete repositories with metadata display
  • Metrics Visualization: Interactive charts (Recharts) showing complexity trends, code churn, and risk analysis
  • Test Prioritization: Recommendations and priority ranking for test cases
  • Bug Report Management: Create, track, and manage bug reports with status workflows
  • Settings: User profile management and GitHub token configuration
UI Standards:
  • Built with Radix UI components for accessibility (WCAG 2.1 Level AA)
  • Styled with Tailwind CSS 3.4.19 for responsive design
  • Keyboard navigation shall be fully supported
  • Color contrast ratios shall meet accessibility standards

4.2 Functional Requirements

Each requirement uses IEEE 29148 "shall" language and includes unique identifiers for traceability.

4.2.1 User Authentication & Management

  • FR-01: The system shall allow new users to register by providing a username, email address, and password.
  • FR-02: The system shall validate that the username and email are unique before creating a user account.
  • FR-03: The system shall hash user passwords using PBKDF2-SHA256 (via werkzeug.security) before storing them in the database.
  • FR-04: The system shall authenticate users via username/password and create a server-side session upon successful login.
  • FR-05: The system shall allow authenticated users to update their profile information (full name, organization).
  • FR-06: The system shall allow users to store a GitHub Personal Access Token securely in the Settings table for repository access.

4.2.2 Repository Management

  • FR-07: The system shall allow authenticated users to add a GitHub repository by providing a valid repository URL.
  • FR-08: The system shall fetch repository metadata (name, owner, description, language, stars, forks) from the GitHub API upon repository addition.
  • FR-09: The system shall display a paginated list of all repositories owned by the authenticated user.
  • FR-10: The system shall allow users to delete a repository and cascade-delete all associated metrics, predictions, bug reports, and test cases.

4.2.3 Code Analysis & Metrics

  • FR-11: The system shall clone the selected GitHub repository to the server for analysis.
  • FR-12: The system shall identify and extract all analyzable source code files (Python, JavaScript, TypeScript, Java) from the cloned repository.
  • FR-13: The system shall calculate the following metrics for each source file: Lines of Code (LOC), Cyclomatic Complexity (CC), Maintainability Index (MI), Halstead Volume, and Halstead Difficulty.
  • FR-14: The system shall use Radon for Python file analysis and Lizard for multi-language code complexity computation.
  • FR-15: The system shall calculate code churn (number of commits modifying each file) using GitPython.
  • FR-16: The system shall persist all calculated metrics in the CODE_METRIC table with a timestamp.

4.2.4 Machine Learning Predictions

  • FR-17: The system shall train a Random Forest classifier using scikit-learn on code metric features to predict fault-prone modules.
  • FR-18: The system shall extract feature vectors from code metrics (LOC, CC, MI, Halstead metrics, code churn) for each file.
  • FR-19: The system shall output a fault probability (0.0 – 1.0) and a confidence score (0.0 – 1.0) for each analyzed file.
  • FR-20: The system shall classify each file into a risk level: Low (0–0.3), Medium (0.3–0.6), High (0.6–0.8), or Critical (0.8–1.0) based on fault probability.
  • FR-21: The system shall persist trained model artifacts as .pkl files using Joblib for later inference.

4.2.5 Dashboard & Visualization

  • FR-22: The system shall display an overview dashboard with summary cards showing: Total Files Analyzed, High Risk Modules Count, Average Complexity, and Recent Analysis Timestamp.
  • FR-23: The system shall display a risk distribution chart (pie/donut) showing the breakdown of Low, Medium, High, and Critical risk files.
  • FR-24: The system shall display a complexity trend line chart showing cyclomatic complexity changes over analysis history.
  • FR-25: The system shall display a list of the 5 most recently analyzed repositories with timestamps.

4.2.6 Test Prioritization

  • FR-26: The system shall generate test case prioritization recommendations based on ML-predicted risk scores.
  • FR-27: The system shall display test cases ranked by priority score in descending order.
  • FR-28: The system shall indicate the target module/file associated with each test case recommendation.

4.2.7 Bug Report Management

  • FR-29: The system shall allow authenticated users to create bug reports with fields: title, description, severity (Low/Medium/High/Critical), and associated file path.
  • FR-30: The system shall support a bug lifecycle with status transitions: Open → In Progress → Resolved → Closed.
  • FR-31: The system shall display all bug reports for a repository with filtering by status and severity.

4.2.8 Recommendations & Settings

  • FR-32: The system shall generate code improvement recommendations based on detected complexity and risk patterns.
  • FR-33: The system shall allow users to configure: theme (light/dark), notification preferences, auto-analyze toggle, and analysis depth level.

4.3 Non-Functional Requirements

4.3.1 Performance

  • NFR-01: The system shall respond to 95% of API requests within 2 seconds under normal load.
  • NFR-02: The Dashboard page shall load within 3 seconds including chart rendering.
  • NFR-03: Repository analysis shall complete within 5 minutes for repositories containing fewer than 1,000 source files.
  • NFR-04: The system shall support at least 50 concurrent authenticated users.
  • NFR-05: The database shall handle a minimum of 100 queries per second.

4.3.2 Reliability

  • NFR-06: The system shall gracefully handle GitHub API failures without crashing, displaying appropriate error messages.
  • NFR-07: Database errors shall not corrupt existing data; transactions shall be rolled back on failure.
  • NFR-08: Failed repository analyses shall not corrupt existing metrics in the database.
  • NFR-09: All user inputs shall be validated with appropriate error messages for invalid data.

4.3.3 Availability

  • NFR-10: The system shall maintain 99% uptime during business hours, excluding planned maintenance.
  • NFR-11: Scheduled maintenance windows shall not exceed 2 hours.

4.3.4 Security

  • NFR-12: User passwords shall be hashed using PBKDF2-SHA256 — plain-text passwords shall never be stored.
  • NFR-13: User sessions shall expire after 7 days of inactivity.
  • NFR-14: All protected API endpoints shall require valid authentication; unauthenticated requests shall receive HTTP 401.
  • NFR-15: Users shall only access their own repositories, metrics, and data.
  • NFR-16: SQL injection shall be prevented through ORM parameterized queries (SQLAlchemy).
  • NFR-17: Cross-site scripting (XSS) shall be prevented via React's automatic output escaping.
  • NFR-18: GitHub tokens shall be encrypted before storage and never logged or exposed in API responses.

4.3.5 Maintainability

  • NFR-19: The codebase shall follow established coding standards: PEP 8 for Python, ESLint for TypeScript.
  • NFR-20: The system shall use a layered architecture with loosely coupled components.
  • NFR-21: Code coverage shall exceed 70% for backend services.
  • NFR-22: All public functions and API endpoints shall be documented.

4.3.6 Portability

  • NFR-23: The frontend shall function correctly on Chrome 100+, Firefox 100+, Safari 15+, Edge 100+.
  • NFR-24: The backend shall run on Windows 10/11, Ubuntu 20.04+, and macOS 11+ without modification.
  • NFR-25: The UI shall be responsive across screen widths from 768px to 1920px.



























4.4 Database Requirements

Database Type: SQLite 3.35+

File-based relational database suitable for single-instance deployment with cross-platform portability.

Data Integrity & Constraints:

  • Primary keys: All tables shall have unique auto-incrementing integer primary key (id)
  • Foreign keys: Referential integrity shall be maintained between all related tables
  • Unique constraints: username, email in USER table shall be unique
  • Indexes: Shall be created on frequently queried columns (user_id, repository_id, file_path)
  • Cascading deletes: Deleting a repository shall cascade to metrics, predictions, bugs, and test cases
